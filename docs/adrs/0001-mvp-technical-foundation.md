# ADR-001: Технологический фундамент MVP LLM-ассистента

- **Статус:** Accepted
- **Дата:** 2026-05-23
- **Контекст:** [idea.md](../idea.md), [vision.md](../vision.md)

## Контекст

Необходимо проверить идею LLM-ассистента в формате Telegram-бота (см. [idea.md](../idea.md)).  
Цель MVP — максимально простое решение без оверинжиниринга: диалог с пользователем и ответы на вопросы в заданной роли.

## Решение

### 1. Технологический стек

| Область | Решение |
|---------|---------|
| Язык | Python **3.11** |
| Зависимости | **uv** (`pyproject.toml` + `uv.lock`, секция `[project]`) |
| Telegram | **aiogram 3.x**, метод **polling** (без webhook) |
| LLM | **openai**-клиент → провайдер **OpenRouter** |
| Модель | `nvidia/nemotron-3-nano-30b-a3b:free` (задаётся в конфиге) |
| Конфигурация env | **python-dotenv** (загрузка из `.env`) |
| Контейнер | **Docker** — только локальный запуск |
| Автоматизация | **`make.sh`** + **`Makefile`** (обёртка) |

**Зависимости MVP:** `aiogram`, `openai`, `python-dotenv`.  
БД, очереди, web-фреймворки на этапе MVP не используются.

### 2. Сборка и запуск

- Вся логика команд — в **`make.sh`**
- **`Makefile`** — тонкая обёртка, делегирует вызовы в `make.sh`
- Команды MVP: `install`, `run`, `docker-run`
- Примеры: `make run` / `./make.sh run`, `make docker-run` / `./make.sh docker-run`
- На Windows: предпочтительно **WSL**; `make.sh` также работает в Git Bash
- Docker запускается **через WSL** (Docker Desktop + WSL2)
- Деплой на удалённый сервер **не предусмотрен**

### 3. Принципы разработки

**KISS:**
- MVP реализует цепочку: пользователь → Telegram → LLM → ответ
- История диалога — **в памяти процесса** (сбрасывается при перезапуске)
- На пользователя хранятся **последние 20 пар** сообщений user/assistant

**ООП: 1 класс = 1 файл:**
- Классы в `src/`, имя файла — snake_case от имени класса
- Зависимости через конструктор; сборка — в `main.py`

**Классы MVP:**

| Класс | Ответственность |
|-------|-----------------|
| `Config` | Чтение настроек из env |
| `Bot` | Приём и отправка сообщений Telegram |
| `LlmClient` | Запросы к OpenRouter |
| `DialogService` | История диалога и вызов LLM |

**Async:** весь код асинхронный (`async`/`await`) — требование aiogram 3.

**Ограничения:**
- Нет абстрактных базовых классов и паттернов «на будущее»
- Нет тестовой инфраструктуры на старте

## Последствия

### Положительные

- Минимальный стек — быстрый старт и простая отладка
- uv ускоряет управление зависимостями
- polling и in-memory история не требуют инфраструктуры
- `make` + `make.sh` покрывают Linux, macOS и Windows (WSL)

### Отрицательные / ограничения

- История диалога теряется при перезапуске бота
- polling менее масштабируем, чем webhook (приемлемо для MVP)
- Docker на Windows зависит от WSL2
- Бесплатная модель OpenRouter может иметь лимиты по скорости и доступности
